System Architecture

End-to-end data flow from 19 public threat feeds through the Python parsers, SQLite WAL database, Flask web application, and dark-themed Bootstrap 5 browser UI. Includes component diagram.

  • fetcher.py — domain/URL HTTP parsers
  • ip_fetcher.py — IPv4 HTTP parsers
  • SQLite DB — WAL mode, domain_base + ip_subnet columns
  • Flask app — 10 routes, JSON API
Data Normalisation & Cross-Reference

Two-stage domain normalisation using the Public Suffix List (PSL) and /24 subnet grouping for IPs. Cross-source agreement engine with 2+, 3+, 4+ confidence tiers.

  • PSL: subdomain → registered domain_base
  • /24 grouping: exact IP → subnet confidence
  • Before/after comparison panels in UI
  • All original data preserved at full resolution
Active iptables Integration (Concept)

First-packet trigger model: DROP rules are installed on-demand only when a connection to a known-malicious destination is first attempted — not pre-loaded from the full database.

  • First SYN exits (no prior knowledge required)
  • Good & Bad DB lookup in parallel
  • DROP rule installed immediately if malicious
  • No handshake completes — no session established
  • ipset with TTL prevents unbounded rule growth
DNS Danger Extension (Concept)

A structured TXT record carrying danger status, confidence, and source attribution. Major resolvers (Google, Cloudflare) cache and propagate the field globally via normal DNS TTL without requiring any modification.

  • Backwards-compatible: ignored by non-aware resolvers
  • Propagates to billions of clients within minutes
  • Short TTL (5–15 min) for fast threat revocation
  • Feed attribution for trust weighting by clients
Hardware Requirements

Demand-driven enforcement keeps the active ruleset small regardless of database size. No pre-loading of 500k+ entries into the kernel.

  • Home/SOHO: Raspberry Pi 4, 2 GB RAM, 16 GB SD
  • Small office: any x86 mini-PC or VM, 4 GB RAM
  • Enterprise: VM on existing hypervisor, 8 GB RAM
  • No cloud required — fully self-hosted
